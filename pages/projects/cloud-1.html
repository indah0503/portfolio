Project I · AWS Cloud Architecture

3-Tier Architecture

Amazon VPC  ·  EC2 · RDS · Security Groups · NAT Gateway · Multi-AZ

Phase A · Network Configuration

VPC, Subnets & Routing

Amazon VPC Elastic IP Internet Gateway NAT Gateway Route Tables Security Groups
01Created a VPC with CIDR 192.168.0.0/16 — serves as the isolated network boundary for all three tiers.
02Provisioned 4 subnets: Public Subnet (192.168.1.0/24) for the User Tier, and Private Subnet 1–3 for App and Data tiers. Private Subnet 3 uses a separate AZ for Multi-AZ fault tolerance.
03Allocated an Elastic IP and attached an Internet Gateway to the VPC — enables outbound internet access for the public subnet.
04Deployed a NAT Gateway in the public subnet using the Elastic IP — allows private subnet instances to reach the internet without being publicly exposed.
05Created two route tables: public table routes 0.0.0.0/0 to the Internet Gateway; private table routes 0.0.0.0/0 to the NAT Gateway. Each subnet was associated with its respective table.
06Defined 4 Security Groups — BastionHost-SG, WebServer-SG, AppServer-SG, and DB-SG — with least-privilege inbound rules referencing each other's SG as source instead of open CIDR ranges.
Network topology established — all tiers isolated and routed correctly

Phase B · Compute Configuration

EC2 Instance Deployment

Amazon EC2 Amazon Linux 2023 Bastion Host Apache / LAMP MariaDB User Data
01Launched Bastion Host (t3.micro) in the public subnet — acts as a secure jump server to reach private instances via SSH. Auto-assign public IP enabled.
02Launched Web Server (t3.micro) in the public subnet with a User Data bootstrap script that installs httpd, the LAMP stack, and enables Apache on startup.
03Launched App Server (t3.micro) in Private Subnet 1 — no public IP assigned. User Data bootstraps mariadb105-server and starts the service automatically.
3 instances running — 2 public, 1 private

Phase C · Data Tier

RDS Database with Multi-AZ

Amazon RDS MySQL / Aurora DB Subnet Group Multi-AZ Private Subnet
01Created a DB Subnet Group spanning Private Subnets 3 and 4 across two Availability Zones — prerequisite for Multi-AZ RDS deployment.
02Provisioned an RDS MySQL instance assigned to the DB Subnet Group, with the DB-SG restricting inbound traffic exclusively to App Server and Bastion Host security groups.
03Multi-AZ configuration ensures automatic failover to a standby replica in the second AZ if the primary instance becomes unavailable — no manual intervention required.
Database deployed — highly available across two AZs

Phase D · Connectivity Testing

End-to-End Validation via PuTTY

PuTTY / SSH pscp Bastion Tunneling ping MySQL CLI
01Connected to Bastion Host via PuTTY using its public IP — validated SSH access from external network.
02Transferred the .pem key file from local Windows machine to the Bastion Host using pscp over port 22 — needed for onward SSH into private instances.
03From the Bastion Host, SSH'd into the App Server using the transferred .pem file — confirmed private subnet reachability through the jump host.
04Ran ping to the Web Server's private IP from the App Server — validated cross-tier ICMP connectivity allowed by the Security Group rules.
05Established a MySQL connection from App Server to the RDS endpoint — confirmed port 3306 access is open between the App Server and Database tiers.
All tiers connected — architecture fully validated
